## 第一部分：项目全景

### (a) 三个项目概览

**free-code（Claude Code分支）。** free-code是Anthropic的Claude Code——面向Claude模型家族的、生产级终端原生AI编程Agent——的一个去除遥测、移除护栏的分支版本。作为一个运行在Bun上的单体TypeScript应用（约1,934个源文件），它提供了约50个工具，涵盖文件系统操作、Shell执行、git管理、LSP集成、MCP协议支持以及多Agent编排。其架构核心是`QueryEngine`类（1,177行），通过异步生成器管道管理Agent的核心生命周期，UI层使用Ink（React for terminals）。free-code支持五个LLM提供商（Anthropic、OpenAI Codex、AWS Bedrock、Google Vertex和Anthropic Foundry），包含88个功能标志——其中54个编译通过——使其成为实验的激进平台。最初由Anthropic开发，并于2026年3月31日通过npm source-map暴露重构而来，free-code分支去除了遥测功能，剥离了硬编码的安全护栏，并解锁了所有实验性功能。

**OpenCode。** OpenCode是一个自称为"Agentic编码工具"的开源项目，定位为Devon和GitHub Copilot的终结者（"the terminal for the agentic coding era"）。在架构上，它采用18个包的monorepo结构，使用Bun运行，核心是用Effect-TS构建的——具有强类型服务、依赖注入和事件驱动解耦的函数式TypeScript。OpenCode支持15+个LLM提供商，提供17+个工具，包括LSP集成（goToDefinition、findReferences、hover、diagnostics）、AST-grep（基于解析树的代码搜索和重写）、WebFetch、WebSearch、TodoWrite、Task和Question。其最鲜明的架构特征是基于Effect的流式管道、每个目录拥有独立总线和文件系统抽象的`InstanceState`模式，以及一个新兴的V2事件溯源对话格式。

**Pi Agent Harness。** Pi是一个"智能、快速且轻量的编程Agent"，单次npm安装即可使用，设计为可嵌入到CI/CD管道中。在架构上，它采用5个包的npm workspaces monorepo，核心是强类型（通过TypeBox）且对LLM提供商无偏见（通过`packages/ai`抽象支持22+个提供商）。Pi力求最小化——仅7个内置工具（Read、Bash、Edit、Write、Grep、Find、Ls）——并依赖扩展模块和用户自定义配置来获取额外功能。它独特的基于JSONL的会话格式设计为可通过Hugging Face共享，其树形结构的分支模型（parentId链）天然支持会话克隆和实验。

#### 跨项目工具规模

| 项目 | 工具数量 | 关键工具 | 扩展机制 |
|---------|-----------|-----------|-------------------|
| free-code | ~50 | Bash、Read、Write、Edit、Grep、Glob、LSP、MCP、Task、Agent | Skills、Plugins、MCP |
| OpenCode | 17+ | Bash、Read、Write、Edit、Grep、Glob、LSP、TodoWrite、Task、Question、WebFetch、WebSearch | Plugin SDK、MCP |
| Pi | 7 | Read、Bash、Edit、Write、Grep、Find、Ls | TypeScript Extension Modules |

工具数量的差异（7到100+）反映了关于多少功能内建与多少功能留给扩展机制的不同哲学。但模式是相同的：**工具就是LLM的手**。

### (b) 交叉模式

尽管独立起源且设计哲学各不相同，三个项目共享六个基本的交叉模式，这些模式高度趋同。

#### 1. 工具作为LLM与外部状态之间的接口

每个项目都通过工具系统向LLM暴露确定性操作。该模式是：

- **Agent运行时**（Layer 2）维护一个工具注册表——名称到函数的映射。
- **LLM**（Layer 3）将函数调用输出为结构化文本（`tool_use`块、函数调用或XML标签）。
- **Agent运行时**拦截这些调用，执行相应的函数（可能在检查权限之后），并将结果返回给LLM。
- 该循环持续进行直到LLM产生非工具输出或达到轮次预算。

#### 2. 上下文窗口管理

三个项目都投入了大量工程精力来管理进入LLM上下文窗口的内容。当代模型提供200K–1M的token窗口，长时间的编码会话很容易溢出。共享模式是一个多层策略：

- **裁剪/截断**：单个工具输出受字符/行数限制（free-code：grep为2000字符，OpenCode：2000字符，文件50KB）。完整输出存储在外部。
- **压缩/摘要**：当总上下文接近窗口限制时，较旧的对话轮次通过单独的LLM调用被压缩为摘要。所有三个项目都实现了LLM驱动的摘要化。
- **溢出检测**：持续监控上下文使用量。当token超过`(窗口 - 保留输出空间 - 安全缓冲区)`时，压缩自动触发。

这不是一个便利功能——它是一个生存必需。没有上下文管理，Agent无法处理非小规模的项目。

#### 3. 会话和状态追踪

三个项目都在LLM上下文窗口之外维护持久的会话状态。会话是持久的、可恢复的对象，能够在进程重启后继续存在：

- **free-code**：每个对话的`QueryEngine`实例，包含可变消息数组、成本跟踪和git状态缓存。会话钩子提供生命周期观察点。
- **OpenCode**：基于SQLite/Drizzle的会话持久化，包含结构化消息存储、会话回滚能力，以及新兴的V2事件溯源架构（用于多实例同步）。
- **Pi**：仅追加的JSONL会话文件，带有树形结构分支（parentId链）、压缩摘要和分支摘要。会话设计为可通过Hugging Face共享。

会话是Agent的**外部化记忆**——在任何单独的LLM调用或上下文窗口之外持续存在的状态。

#### 4. Bash/Shell接口作为主要工具

在每个项目中，bash/shell工具是继文件读/写之后最重要的工具。它作为通用后备：任何没有专用工具的操作都可以通过Shell命令执行。这不仅便利——它反映了深刻的架构洞见：

- LLM在Shell命令上进行了预训练，天然地以`grep`、`find`、`git`、`npm`的方式"思考"。Shell接口直接映射到其训练分布。
- Shell命令是可组合的（`grep | sort | head -5`），使得在单次调用中实现复杂的信息检索成为可能。
- Shell输出是确定性的（模文件系统状态）、有界的，并且自文档化（命令字符串本身就是查询）。
- Shell具有状态性但不携带状态：`git log --oneline -5`提供git历史的快照，而无需LLM跟踪完整的提交DAG。

Free-code的架构明确表达了这一点："Bash是不可约减的核心。每一个其他工具都是对Bash的优化。"OpenCode同样指出，即便是grep和glob等更高级工具也"最终通过类Shell的执行模型调用ripgrep。"

#### 5. 任务分解（显式或隐式）

三个项目通过某种机制将大型任务分解为更小的单元：

- **free-code**：通过`AgentTool`创建子Agent，每个子Agent拥有独立的上下文窗口。`TaskTool`创建带有状态跟踪的结构化任务项。实验性的"Agent Swarms"功能实现并行多Agent协调。
- **OpenCode**：`todowrite`工具在上下文内提供显式的任务状态跟踪。子Agent可作为独立会话创建。primary/subagent层次结构允许并行工作。
- **Pi**：任务分解是隐式的而非显式的——通过会话树分支（parentId链）处理，而非结构化任务项。其哲学是"用户可以要求pi自行构建他们想要的东西。"

尽管实现不同，模式是通用的：**单个LLM调用无法解决一个大型项目；工作必须分解到多个调用、多个Agent或多个会话中。**

#### 6. 权限和安全模型

三个项目都实现了权限系统，以控制LLM可以使用哪些工具：

- **free-code**：集成的允许/拒绝/询问规则，带有绕过和计划模式。每个工具调用经过`canUseTool()`。钩子拦截器实现自定义权限逻辑。
- **OpenCode**：基于规则的权限系统，带有通配符模式匹配和Shell命令的tree-sitter AST扫描。用户通过基于Bus的交互式许可获得批准。权限可以限定到每个目录的InstanceState。
- **Pi**：扩展模块注册可以转换或拒绝工具调用的自定义钩子。权限模型不像其他三个那样正式化，这与pi的最小主义一致。

### (c) 分叉路径

虽然趋同模式引人注目，但三个项目在几个维度上存在显著分歧：

#### 单体 vs. Monorepo

- **单体架构**：free-code（单个`src/`目录，Bun原生打包）。这些优化了部署简单性，但牺牲了模块化和贡献者可访问性。
- **Monorepo架构**：OpenCode（18个包，Turborepo编排）和Pi（5个包，npm workspaces）。这些优化了模块化、可测试性和贡献者上手体验，但牺牲了构建复杂性。

单体方法体现了由模型提供商（Anthropic for free-code）构建的、以用户体验为优化目标的工具特征。Monorepo方法体现了以生态系统增长为优化目标的开源平台（OpenCode、Pi）的特征。

#### 提示工程 vs. 编程式循环

- **重提示**：free-code广泛依赖系统提示构建——CLAUDE.md文件、技能描述、MCP资源列表、环境信息和工具定义全部预加载到上下文中。假设LLM在获得足够上下文的情况下会做出正确决策。
- **编程式**：OpenCode使用Effect-TS将Agent执行循环建模为有限状态机，具有类型化服务、依赖注入和事件驱动解耦。假设正确性应由运行时而非提示来强制执行。
- **最小主义**：Pi依赖一个紧凑的系统模板和transform钩子。假设复杂性应归属于扩展而非核心。

#### 状态管理策略

| 项目 | 状态存储 | 持久化格式 | 关键机制 |
|---------|--------------|-------------------|---------------|
| free-code | 内存（AppState store）+ 文件系统 | 类Redux的可观察存储 | 每个对话的QueryEngine，带有可变消息数组 |
| OpenCode | SQLite/Drizzle + 事件总线 | 结构化消息表 + V2事件溯源 | 每个目录的InstanceState，带有ScopedCache |
| Pi | 文件系统（JSONL） | 仅追加JSONL，带有parentId链 | 树形结构会话，带有压缩摘要 |

状态管理的分歧是最具架构意义的。Free-code和OpenCode使用数据库支持的持久化与结构化模式，而Pi使用扁平的JSONL文件以追求透明性和可共享性。

#### LLM提供商策略

- **与提供商绑定**：free-code深度集成Anthropic API格式（`tool_use`块、`BetaContentBlockParam`）。支持其他提供商需要翻译层。
- **多提供商设计**：OpenCode通过Vercel AI SDK支持15+个提供商，并配有特定于模型的系统提示。
- **类型安全的多提供商**：Pi的`packages/ai`层使用TypeBox进行模式验证，支持22+个提供商，并具有流式类型安全。

三个项目的趋势都是朝着**提供商独立性**发展——free-code与Anthropic的紧密耦合是例外，反映了它作为一个配套工具而非平台的起源。

## 第二部分：马尔可夫分解论题

### (d) 假设陈述

本研究的核心假设是：

> **"Agent framework的意义是不是就在于，对非马尔可夫过程的大型项目，或内容库解缠，把它尽量变成一个纯马尔可夫状态，用类bash的方式提供，方便gpt型lm这种纯马尔可夫状态机去操作。"**

该假设包含三个不同的主张：

1. **LLM是马尔可夫状态机**——它们的下一个token预测仅取决于当前的上下文窗口（状态），而不取决于该窗口之外的任何先前交互历史。
2. **真实世界的软件项目是非马尔可夫的**——项目的状态（代码库、git历史、构建产物、问题跟踪器、团队知识）太大了，无法适应任何实际的上下文窗口，有效的行动需要知晓决策、变更和理由的完整历史。
3. **Agent框架弥合了这一差距**——它们通过工具媒介的状态访问、上下文压缩以及作为通用桥梁的类bash接口，将非马尔可夫的项目现实转化为马尔可夫的上下文切片。

### (e) 理论基础

#### 形式化证明：LLM是马尔可夫链

该假设的理论基础建立在Zekri等人（2024）的形式化证明之上：自回归语言模型在数学上是马尔可夫链。证明是直截了当的：

1. 自回归LLM计算`P(t_n | t_1, t_2, ..., t_{n-1})`——给定所有先前token的下一个token的概率。
2. 模型在任意点的"状态"是上下文窗口中所有token的连接：`S = [t_1, t_2, ..., t_{K}]`，其中`K`是上下文窗口大小。
3. 下一个输出**仅**取决于此状态`S`，而不取决于已超出上下文窗口的token或任何外部记忆。
4. 这满足马尔可夫性质：`P(S_{t+1} | S_t, S_{t-1}, ..., S_1) = P(S_{t+1} | S_t)`。

用形式化术语来说，自回归LLM是一个**马尔可夫链**，其状态空间是长度≤`K`的所有可能token序列的集合。模型本身在调用之间是无状态的——每次推理都从提供的上下文窗口从头开始。

#### 状态空间爆炸问题

马尔可夫性质伴随着一个毁灭性的实际约束：状态空间大小。对于词汇量大小为`T`、上下文窗口为`K`的LLM，状态空间的基数为`O(T^K)`。即使对于适度的值（T ≈ 50,000，K ≈ 200,000），这个数字也是天文数字般巨大——远远超出任何训练数据所能覆盖的范围。

这意味着**LLM理论上可以进入的大多数状态在训练中从未出现过**。LLM在其可能状态空间的一个无穷小子空间内运行——即类似于其训练分布的token序列子空间。这就是提示工程有效的原因：系统提示将LLM约束到状态空间中经过良好训练的区域。

#### "非马尔可夫"在此语境下的含义

如果一个过程的下一个状态不仅取决于当前状态——还取决于我们如何到达当前状态的历史，则该过程是**非马尔可夫的**。在软件工程中：

- 一个包含100,000个文件的代码库无法放入任何上下文窗口。状态太大了。
- 一个函数为何以某种方式编写，取决于六个月前的提交、issue线程中的设计讨论以及默会的团队知识。
- 下一个正确的变更取决于构建配置、测试结果、依赖版本以及数千个相互依赖的组件之间的交互。
- 一个bug可能是由文件A、B和C之间的交互引起的——但发现这种交互需要阅读所有三个文件、理解它们的关系并追踪执行路径。

所有这些都是**根本性的非马尔可夫**：做出下一个正确决策所需的信息分布在一个比任何LLM上下文窗口都要大得多的状态空间中。

#### 根本性的阻抗失配

这在以下两者之间产生了一种**根本性的阻抗失配**：

- **LLM**：一个马尔可夫状态机，只能对当前上下文窗口中的内容进行推理。它在调用之间无法"记住"任何东西。它的世界是有界的、有限的、自包含的。
- **项目**：一个非马尔可夫系统，其中下一个决策所需的信息分布在上百万token的代码、数千次提交的历史以及外部状态（CI结果、团队讨论、文档、运行时行为）之中。

Agent框架是**阻抗匹配层**——该组件将项目的无界的、历史的、分布式的现实，转化为马尔可夫LLM可以消费的有界的、自包含的上下文切片。

### (f) 来自三个项目的证据

每个项目都提供了具体的、可验证的证据，表明它实现了马尔可夫分解。证据是架构层面的，而非推测性的——它来自对源代码结构、命名模块和算法实现的分析。

#### free-code：压缩作为马尔可夫状态压缩

Free-code实现了一个**三层压缩管道**，逐步将对话历史转换为马尔可夫状态：

1. **微压缩**（`microCompact.ts`）：在工具结果进入消息历史之前缩减它们。对于grep、read和glob输出，内容被token截断或替换为结构化摘要。对于Shell输出，多余输出被裁剪并附加注释：`"... [X lines of output truncated]"`。这是第一次马尔可夫转换——无界的世界输出成为一个有界的上下文条目。

2. **自动压缩**（`autoCompact.ts`）：当总上下文token超过窗口的70%时，触发完整的对话压缩。一个Haiku模型将整个对话历史总结为结构化摘要。原始消息被替换为摘要加上最近的轮次。这将数千个token的对话历史转换为约500个token的马尔可夫摘要。

3. **裁剪压缩**（`snipCompact.ts`）：对于极长的会话（SDK/无头调用者），在标识边界处激进地裁剪历史，在UI回滚中保留完整历史的同时呈现压缩视图。这是最激进的马尔可夫转换——大块历史被替换为边界标记。

`context.ts`模块每轮从CLAUDE.md文件、git状态、环境信息、工具定义和技能描述中构建系统提示部分——这是一个**每轮状态切片**，而非全局项目视图。

**关键证据**：上下文窗口中的工具结果不是原始输出——它是微压缩后的表示。完整输出存在于磁盘上（供后续参考），但LLM只看到压缩版本。这是马尔可夫分解的实际运行：无界的文件读取成为一个有界的、可操作的摘要。

#### OpenCode：结构化压缩和实例作用域

OpenCode的马尔可夫分解方法是三个项目中最正式结构化了的：

1. **压缩作为结构化摘要**：压缩模板使用显式的分节——目标、约束与偏好、进度（已完成/进行中/阻塞）、关键决策、下一步、关键上下文和相关文件。这种结构化格式确保压缩后的上下文保持**可操作性**——它精确地保留了下一个决策所需的信息。

2. **三层上下文策略**：裁剪（2000字符工具输出限制）→ 压缩（LLM驱动的摘要化，`PRUNE_MINIMUM: 20,000`、`PRUNE_PROTECT: 40,000`、`DEFAULT_TAIL_TURNS: 2`，以及精确的保留近期token范围）→ 截断（2000行/50KB硬限制）。

3. **实例作用域**：`InstanceState`模式创建每个目录的自包含世界，每个世界拥有自己的总线、LSP服务器、权限规则、Shell环境和文件系统抽象。这种作用域在项目之间创建了马尔可夫边界——一个项目的状态无法泄露到另一个项目中。

4. **溢出检测**：`overflow.ts`模块计算可用上下文为`max(0, model.limit.input - reserved)`，其中`reserved = min(COMPACTION_BUFFER, maxOutputTokens)`。这是马尔可夫状态的一个精确数值边界。

5. **V2压缩对话格式**：新兴的事件溯源架构以压缩格式序列化对话，将状态与呈现分离，实现高效的上下文重建。

**关键证据**：压缩摘要模板被明确设计为**对未来的决策足够的**。结构化格式（目标、进度、关键决策、下一步）并非随意——它是使下一个LLM轮次变得马尔可夫（自包含且充分）的最小信息集。

#### Pi：自动压缩和分支摘要

Pi以最大的透明度和最小的机制实现马尔可夫分解：

1. **自动压缩**：当`contextTokens > contextWindow - reserveTokens`（默认保留：16,384）时，压缩自动触发。与其他项目不同，pi使用**迭代摘要化**——如果摘要本身超出token限制，它会被再次摘要化，递归进行，直到上下文适配。

2. **切点逻辑**：`findValidCutPoints()`函数扫描对话条目以找到安全的压缩边界。有效切点是`user`、`assistant`、`custom`、`bashExecution`、`branchSummary`和`compactionSummary`消息。工具结果永远不会被切割（它们必须跟随其工具调用）。当在包含工具调用的assistant消息处切割时，工具调用及其结果作为一个单元被保留。

3. **分支摘要**：在导航会话树分支（parentId链）时，会生成分支特定的摘要。这使LLM能够在不同分支中"从上次中断的地方继续"，而无需重放完整历史。

4. **`convertToLlm`边界**：`transformContext`钩子和`convertToLlm`函数在Agent的内部表示与LLM的输入格式之间形成显式边界。Agent消息在进入LLM之前被转换、裁剪和压缩——这是马尔可夫转换边界。

5. **JSONL会话文件**：通过将会话存储为仅追加JSONL并带有树形结构，pi将完整的非马尔可夫历史外部化，同时仅在上下文中保留马尔可夫切片。

**关键证据**：迭代摘要化方法是递归马尔可夫分解的最纯粹实现。当摘要的摘要溢出时，它会被再次摘要化——系统将递归压缩直到状态适配。这直接实现了核心理念：通过连续的压缩将非马尔可夫历史转换为马尔可夫状态。

### (g) 反对简化假设的证据

尽管三个项目都提供了强有力的证据表明马尔可夫分解是一个核心设计关注点，但假设的**简化版本**——即Agent框架仅仅将非马尔可夫过程转换为马尔可夫过程——是不充分的。以下几方面的证据主张需要更细致的理解：

#### 1. 任务分解不仅仅是状态转换

Agent框架所做的不仅仅是转换状态。它们**分解任务**。这在根本上不同于状态转换：

- free-code和OpenCode中的子Agent会创建独立的工作者，并行处理子任务。每个子Agent都有自己的上下文窗口、自己的马尔可夫状态和自己的决策循环。这不是状态转换——它是跨多个马尔可夫进程的**工作分配**。
- OpenCode中的待办列表创建了显式的分解结构，这些结构在压缩过程中持续存在。这些不是状态转换——它们是指导马尔可夫决策过程的**任务表示**。

#### 2. Bash接口是一种方法，而非唯一的方法

虽然bash普遍存在，但它并非唯一的接口：

- OpenCode的**LSP集成**使LLM能够执行通过bash不切实际的语义操作（goToDefinition、findReferences、hover、diagnostics）。LSP查询返回结构化的、类型化的信息，与文本流有根本不同。
- OpenCode的**AST-grep**工具使用解析树而非文本正则表达式进行基于模式的代码搜索和重写。这是一个语义接口，而非文本接口。
- Pi的**类型化工具系统**使用TypeBox模式验证工具输入和输出。这超越了bash的非结构化文本模型。

- Free-code的**notebook编辑工具**和**web浏览器工具**提供了bash无法复制的结构化交互模型。

Bash接口是**最小公分母**——普遍可用、被LLM普遍理解、并且普遍可组合。但专用工具为特定领域提供了更高效、更可靠、更结构化的接口。

#### 3. LLM的上下文窗口在理论上是非马尔可夫的

假设指出LLM是"纯马尔可夫状态机"。这在数学层面是正确的（token预测仅取决于上下文窗口），但**上下文窗口的内容**在实践中是非马尔可夫的：

- 上下文窗口中的对话历史包括来自早期轮次的工具调用和结果。这是**情景记忆**——LLM可以看到它之前做了什么以及发生了什么。在一个纯马尔可夫过程中，下一个状态仅取决于当前状态，而不取决于我们是如何到达那里的。
- 系统提示是对每一个决策的**持久的、非马尔可夫的影响**。它在会话开始时加载一次，但影响每一个后续轮次，创建了一个跨越整个会话的隐藏状态依赖。
- OpenCode的权限"always"规则创建了在LLM上下文中不可见的持久状态——一个影响哪些工具可用的非马尔可夫影子状态。


Agent框架并没有消除非马尔可夫性；它**控制**了它。完整的历史仍然存在（在会话文件中、在数据库中），但LLM只看到一个被精心管理的切片。

#### 4. Agent框架通过多种策略管理非马尔可夫性

三个项目揭示了马尔可夫分解是通过一个**策略组合**而非单一机制实现的：

| 策略 | free-code | OpenCode | Pi |
|----------|-----------|-----------|----------|-----|
| 上下文压缩 | 微 + 自动 + 裁剪 | 裁剪 + 压缩 + 截断 | 自动 + 迭代 + 分支 |
| 外部状态存储 | 消息历史在内存中 + 文件系统 | SQLite/Drizzle会话表 | JSONL会话文件 |
| 每轮状态构建 | 每轮的System context + CLAUDE.md | Agent特定提示 + 计划提醒 | 模板 + transformContext钩子 |
| 子进程分解 | AgentTool子Agent | Task + primary/subagent层次结构 | 会话树分支 |
| 工具确定性 | 编辑的精确字符串匹配 | Effect类型化工具执行 | 基于钩子的拦截 |

这些策略中没有一个能单独实现马尔可夫分解。它们的**组合**才创造了马尔可夫推理充分的条件。

#### 5. 真正的贡献：在受管理的观察空间内的结构化动作空间

最深刻的洞见不是Agent转换状态，而是它们在一个**受管理的观察空间**（上下文窗口）内创建了一个**结构化的动作空间**（工具）：

- **工具提供确定性的、幂等的操作**：对同一路径的文件读取始终返回相同的内容。对同一模式的grep始终返回相同的匹配。这种确定性使LLM的动作空间可预测且可靠。
- **上下文管理确保充分性**：压缩、裁剪和每轮构建确保每个LLM轮次恰好拥有它所需的信息——不多不少。
- **其组合创建了一个马尔可夫决策过程（MDP）**：LLM观察一个状态（上下文窗口），选择一个动作（工具调用），收到一个奖励（工具结果），并转换到一个新状态（更新后的上下文）。这是一个覆盖在非马尔可夫世界之上的MDP。

### (h) 修正论题（修订版）

基于所有三个项目的证据，该假设需要修正。修正后的版本是：

> **Agent框架并不将非马尔可夫过程转换为马尔可夫过程。相反，它们通过以下方式在非马尔可夫世界之上创建一个马尔可夫决策过程（MDP）叠加层：**
>
> 1. **提供一个具有确定性、幂等操作的结构化动作空间（工具）。** 每个工具调用是对外部状态的一个明确定义的转换，具有有界的、可预测的输出。LLM从这个空间中选择动作，而无需理解工具实现的全部复杂性。
>
> 2. **通过压缩、摘要化和相关性过滤来管理观察空间（上下文窗口）。** 并非所有项目状态都进入上下文窗口——只有与当前决策相关的切片才进入。Agent设计的艺术在于确定该切片应包含什么。
>
> 3. **将长期状态（文件、git、会话存储）外部化，使LLM的马尔可夫上下文窗口只需要当前决策切片。** 项目和对话的完整历史存在于上下文窗口之外。LLM通过工具按需访问它，从不一次性持有全部。
>
> 4. **使用类bash接口作为LLM的符号推理与外部确定性世界之间的通用、确定性桥梁。** Bash是最小公分母——普遍可用、被LLM自然理解、并且能够表达任何操作。更高级的工具是对此基础的优化。
>
> **LLM仍然是一个马尔可夫状态机。Agent框架创造了使马尔可夫推理在非马尔可夫世界中足以实现有效行动的条件。**

这个修正的论题承认：
- 非马尔可夫性并未被消除——它被外部化、管理并按需访问。
- Agent的角色超出了状态转换，还包括任务分解、并行执行、权限管理和错误处理。
- Bash是通用接口，但特定领域的工具是提高效率和可靠性的优化。

其隐喻不是Agent将世界"转换"为LLM可以理解的东西。隐喻是Agent**创建一个透镜**，LLM通过它可以观察一个否则将对其不可见的世界并采取行动。

#### 每个项目如何映射到三层

**第一层（外部状态）：**
- 所有项目在此处理方式相同：文件系统、git仓库和外部服务是基础事实。没有虚拟文件系统——所有修改都是真实的并且可从外部验证。
- free-code："正确性的最终仲裁者是文件系统。操作是真实的、不可逆的，并且可从外部验证。"
- OpenCode：文件系统工具（`read`、`write`、`edit`）通过`AppFS`抽象操作真实文件系统。
- Pi：7个内置工具全部操作真实的文件系统路径。没有虚拟层。


**第二层（Agent运行时）：**
- free-code：`QueryEngine`类（1,177行）管理Agent生命周期。`services/compact/`包含压缩子系统（6个文件）。`services/api/claude.ts`（3,419行）处理LLM通信。

- OpenCode：`packages/opencode/src`中43个模块涵盖工具、会话、提供商、压缩、权限、LSP、effect、bus——一个全面的运行时实现。
- Pi：`packages/coding-agent/src/core/agent-session.ts`（2,750行）和`packages/coding-agent/src/core/compaction/`（740行）构成运行时核心。

**第三层（LLM上下文窗口）：**
- 所有项目汇聚到同一模式：系统提示 + 工具定义 + 任务状态 + 压缩历史 + 环境信息 + 近期工具结果。差异在于上下文如何构建的方式，而非包含什么内容。
- free-code：每轮构建来自`fetchSystemPromptParts()`和`normalizeMessagesForAPI()`。

- OpenCode：8部分上下文构建，包括特定于模型的提示、Agent定义、计划提醒和构建开关。
- Pi：模板驱动，带有`transformContext`钩子和`convertToLlm`边界。

### (j) Agent循环作为通用模式

Agent循环——核心执行循环——在所有三个项目中几乎完全相同。趋同程度如此之强，以至于它似乎是基于LLM的Agent的**唯一可行架构**：

```
┌─────────────────────────────────────────────────────────────┐
│                     通用AGENT循环                            │
│                                                             │
│  1. 用户输入                                                 │
│     │  （自然语言任务、slash命令、RPC请求）                   │
│     ▼                                                       │
│  2. 模板 / 系统提示构建                                     │
│     │  （预加载上下文：技能、工具、环境、git、任务）          │
│     ▼                                                       │
│  3. 上下文组装                                               │
│     │  （必要时压缩，裁剪工具输出，                          │
│     │   组装最终上下文窗口）                                   │
│     ▼                                                       │
│  4. LLM调用                                                 │
│     │  （流式token，解析响应，检测工具调用）                  │
│     ▼                                                       │
│  5. 响应解析                                                 │
│     │  （文本 → 展示，tool_use → 排队执行）                 │
│     ▼                                                       │
│  6. 工具执行                                                 │
│     │  （检查权限，执行工具，捕获结果）                       │
│     ├── 如果工具结果适配：追加到上下文                        │
│     └── 如果工具结果溢出：微压缩                              │
│     ▼                                                       │
│  7. 结果集成                                                 │
│     │  （将工具结果添加到消息历史）                           │
│     ▼                                                       │
│  8. 压缩检查                                                 │
│     │  （估算token，必要时触发压缩）                         │
│     ├── 低于阈值 → 循环回到步骤4                             │
│     └── 超出阈值 → 压缩 → 循环回到步骤4                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

每个项目都实现了此循环，实现风格有所不同：

- **free-code**：步骤1-8作为异步生成器（`QueryEngine.submitMessage()`），在每个阶段yield以进行UI更新。步骤5-6-7-8作为一个连续循环，直到LLM产生非工具响应（完成检查）或轮次预算耗尽。
- **OpenCode**：步骤1-3作为服务层（Config → Provider → Session → Context）。步骤4-7作为基于Effect的流式管道。步骤5包括带有Bus中介的用户批准的权限评估。步骤8作为带有精确token计算的溢出检测。
- **Pi**：步骤1-3作为带有`transformContext()`钩子的模板构建。步骤4-7作为事件驱动的流式处理，带有两层队列（steering vs. follow-up）。步骤8作为基于阈值的自动压缩，带有迭代递归摘要化。

该循环的普遍性表明它**不是一种设计选择**——它是满足以下约束的唯一结构：LLM接收上下文 → 产生动作 → 动作执行 → 结果进入上下文 → 重复。

### (k) 工具接口作为确定性桥梁

工具接口作为概率性LLM（第三层）与确定性世界（第一层）之间的**确定性桥梁**。这是关键的洞见：工具必须是确定性的，才能使MDP叠加层发挥作用。

#### 为什么确定性至关重要

在马尔可夫决策过程中，转换函数`T(s, a) → s'`应是可预测的。给定状态`s`和动作`a`，下一个状态`s'`应该每次相同（或遵循已知的概率分布）。如果工具是非确定性的——如果相同的文件读取返回不同的内容，或相同的grep返回不同的结果——LLM无法从经验中学习。每个轮次变成对不可预测世界的探索。

三个项目都强制确定性：

- **free-code**：`FileEditTool`使用精确字符串匹配——如果`oldString`匹配多次，工具失败。`GlobTool`按修改时间排序返回文件。这是通过规范实现的确定性。
- **OpenCode**：Effect-TS类型化工具执行确保工具结果遵循类型化契约。权限系统的Shell命令扫描使用tree-sitter进行确定性AST解析——相同的命令始终解析为相同的AST。
- **Pi**：工具通过TypeBox模式返回类型化结果。`edit`工具使用精确字符串匹配。转换管道（`transformContext` → `convertToLlm`）是从Agent状态到LLM输入的确定性函数。

#### Bash作为最小公分母

Bash之所以是通用工具，是因为它满足所有确定性要求：

1. **完备性**：任何操作都可以表示为bash命令。
2. **确定性**：相同命令 → 相同输出（模文件系统状态，这正是Agent试图观察和修改的）。
3. **有界性**：Shell输出可被截断；错误通过退出码显式报告。
4. **LLM原生**：LLM在Shell命令上进行了预训练。这是它们用于系统交互的自然语言。
5. **可组合性**：`grep -r 'function' src/ | sort | head -20`在一次调用中组合了查询、过滤和截断。

然而，三个项目也表明**bash是充分的但非最优的**。专用工具提供优势：
- LSP工具提供需要数十个grep命令才能近似获得的结构化、语义信息。
- AST-grep提供bash/sed无法可靠执行的、基于解析树的转换。
- Notebook编辑工具（free-code）提供文本操作会损坏的、单元级的操作。

其原则是：**bash是通用桥梁；专用工具是针对特定地形的优化桥梁。**

### (l) 上下文工程作为核心学科

三个项目共享的最深刻洞见是**上下文工程是Agent设计的核心学科**。进入上下文窗口的内容决定了LLM能做什么。上下文窗口是系统中最稀缺、最昂贵的资源。

#### 进入上下文窗口的内容

上下文窗口有一个固定的token预算（200K-1M token）。该预算如何分配决定了Agent的能力：

1. **系统提示**：用领域知识预加载上下文。Free-code的CLAUDE.md文件、OpenCode的特定于模型的系统提示和Pi的模板都服务于相同的目的：注入知识，无需LLM通过工具调用去发现它。

2. **工具定义**：每个工具为其名称、描述和参数模式消耗上下文token。权衡：更多工具 = 更多能力但更少的对话空间。Pi的7工具最小主义代表了这一权衡的一端。

3. **对话历史**：每个工具调用和结果消耗上下文。压缩是将此历史转换为紧凑摘要的过程。摘要的质量直接决定未来决策的质量。

4. **当前状态**：待办列表、关键文件、环境信息和git状态提供了当前轮次的马尔可夫状态快照。

5. **保留空间**：所有项目都为LLM的响应保留空间。如果上下文窗口是200K token而LLM需要8K token来响应，那么只有192K token可用于上下文。

#### 上下文策展的艺术

上下文工程不是一个已解决的问题——它是一种**艺术形式**，每个项目以不同方式处理：

- **free-code**：激进的每轮构建，带有git状态缓存。"Agent不会将整个项目预加载到上下文中。相反，它在查询时构建状态。"
- **OpenCode**：结构化压缩模板，显式编码"对未来决策重要的内容"。模板设计就是上下文工程。
- **Pi**：迭代摘要化，将递归压缩直到上下文适配。系统宁愿丢失信息也不愿溢出。

共同的主线：**上下文策展是一个有损压缩问题**。Agent必须决定保留什么、摘要什么和丢弃什么。一个糟糕的决策会丢失关键信息。一个好的决策恰好保留LLM所需的内容。

## 第四部分：实践洞见

### (m) 构建Agent系统的设计原则

基于对三个项目的分析，八条原则作为构建Agent系统的经验教训浮现：

#### 1. 将上下文窗口视为宝贵的、有界的固定空间

上下文窗口在运行时不可扩展。你无法在对话中途从LLM提供商购买更多token。每一个字节的上下文都必须物有所值。原则：
- 系统提示应紧凑且信息密集。
- 工具定义应最小化——LLM不需要看到每个工具的每个参数；只需看到与当前任务相关的那些。
- 对话历史应积极压缩。旧的工具结果应被摘要化，而非保留。
- 微压缩（每个工具结果的修剪）应在结果进入消息历史之前发生，而非之后。

**证据**：每个项目都实现了某种形式的每个工具结果截断。Free-code的microCompact、OpenCode的TOOL_OUTPUT_MAX_CHARS、Pi的prune管道——这些不是优化；它们是生存机制。

#### 2. 使工具具有确定性和幂等性

非确定性工具会创建一个非马尔可夫的动作空间，LLM无法预测其行为的结果。原则：
- 文件读取必须对相同路径返回相同内容（或以明确错误失败）。
- 编辑必须精确应用指定的更改（或如果匹配不明确则以明确错误失败）。
- Shell命令应对相同输入产生相同输出（模Agent自身控制的文件系统状态）。
- 工具失败应为显式失败，而非静默的错误结果。

**证据**：free-code的`FileEditTool`当`oldString`匹配多次时失败。OpenCode通过Effect进行的类型化工具执行确保结果类型安全。Pi的编辑工具使用精确字符串匹配。这不是偏好——它是MDP叠加层能够工作的要求。

#### 3. 将所有持久状态外部化

LLM在调用之间是无状态的。它需要"记住"的一切都必须外部化：
- 会话状态在数据库或文件中（不仅仅在内存中）。
- 任务进度作为待办列表（不仅仅在LLM的"记忆"中）。
- 决策理由作为注释或文档（不仅仅在上下文中）。
- 项目状态作为实际的文件系统（基础事实）。

**证据**：三个项目都将会话状态外部化。OpenCode使用SQLite/Drizzle。Pi使用JSONL文件。Free-code使用内存状态和文件系统持久化。媒介不同，但模式是通用的。

#### 4. 在需要之前就设计好压缩

压缩不是你可以稍后添加的奢侈功能。没有它，Agent在任何非简短的任务上都会失败。原则：
- 在第一次长会话之前实现压缩，而非在第一次溢出错误之后。
- 在真实的对话上测试压缩，而非合成的对话。摘要的质量至关重要。
- 设计压缩摘要格式以保留LLM精确需要的内容：目标、约束、进度、决策、下一步。
- 支持迭代/递归压缩——如果摘要本身溢出，再次压缩它。

**证据**：Pi的迭代摘要化是最好的示例——它将递归压缩直到上下文适配。OpenCode的具有显式分节的结构化摘要模板确保压缩后的上下文保持可操作性。Free-code的裁剪压缩处理超出压缩预算的极长会话。

#### 5. Bash接口充分但非最优——添加特定领域的工具

Bash是通用后备，应始终可用。但专用工具对于特定领域更优越：
- LSP集成用于语义代码理解（OpenCode）。
- AST-grep用于基于解析树的转换（OpenCode）。

- Notebook编辑用于数据科学工作流（free-code）。
- MCP用于无需修改Agent的无限可扩展性（三个项目）。

其原则不是"更多工具更好"——而是"工具应匹配LLM的优势"。一个生成优秀Shell命令的LLM可能不需要专用的grep工具。一个在正则表达式上挣扎的LLM可能从AST-grep中受益。

#### 6. 将Agent逻辑与UI渲染分离

三个项目都将Agent运行时（第二层）与用户界面分离。这使得：
- 来自同一核心引擎的多个接口（终端、web、SDK、RPC）。
- 无需终端即可测试Agent逻辑。
- 将Agent嵌入其他应用程序。
- 用于CI/CD和自动化的无头操作。

**证据**：OpenCode的18包monorepo将`packages/opencode`（引擎）与`packages/console`（TUI）、`packages/app`（web）和`packages/sdk`（编程式）分离。Pi的5包monorepo将`packages/coding-agent`（引擎）与`packages/tui`（终端）和`packages/web-ui`（web）分离。Free-code的`entrypoints/sdk/`提供了对REPL使用的同一`QueryEngine`的无头访问。

#### 7. 从第一天起就规划好多Agent编排

对于单个上下文窗口来说太大的任务需要跨多个Agent分解。从一开始就为此设计：
- 具有可配置上下文继承的子Agent创建。
- 具有结果聚合的并行执行。
- Agent间通信渠道（消息传递、共享状态）。
- 每个Agent独立的权限范围。

**证据**：所有项目都支持某种形式的子Agent调度。Free-code的AgentTool、OpenCode的task/subagent层次结构和Pi的会话树分支都支持多Agent工作流。

#### 8. 权限必须细粒度且可审计

Agent可以访问文件系统、Shell和网络。没有适当的权限，它可能造成不可逆的损害。原则：
- 细粒度：不仅仅是"允许shell"——而是"允许当前仓库中的git命令的shell"。
- 可审计：每个工具调用应记录谁批准了它以及为什么。
- 可升级：危险操作应需要显式的用户确认。
- 可撤销：权限应可在会话中撤销而无需重启。

**证据**：OpenCode的基于规则的通配符匹配，带有Shell命令AST扫描。Free-code的allow/deny/ask规则，带有bypass和plan模式。这些不是事后想法——它们被集成到工具执行路径中。

### (n) 常见陷阱

对三个项目弱点的分析揭示了Agent系统设计中反复出现的陷阱：

#### 上下文窗口溢出

尽管有复杂的压缩系统，三个项目都可能经历上下文溢出：
- **free-code**："压缩是有损的。如果摘要错过了关键细节，Agent可能不可挽回地失去上下文。这是马尔可夫分解链中最薄弱的环节。"
- **OpenCode**："摘要化不可避免地丢失细微差别；重复压缩可能降低决策质量。"
- **Pi**：Token估算使用简单的`chars/4`启发式方法——不感知分词器，对于非英语文本或代码密集型内容可能不准确。

根本问题：**压缩是一种没有形式化误差边界的有损压缩**。无法保证摘要保留了未来决策所需的所有信息。

#### 工具非确定性导致级联错误

非确定性工具可能导致级联故障：
- **free-code**："Bash工具可以执行任意命令，没有运行时、输出大小或副作用的上限。一旦被允许，Agent拥有完全的Shell访问权限。"
- **OpenCode**：插件扩展"可以引入可能无法在工具结果模型中完全捕获的任意状态。"

问题：LLM基于工具结果做出决策。如果这些结果不一致（相同查询，不同答案），LLM就无法形成一致的规划。

#### 压缩不足导致性能下降

压缩不足导致两个问题：
1. LLM的上下文窗口被不相关的历史填满，挤出了当前任务信息。
2. LLM的注意力被数千个token的历史稀释了，降低了决策质量。

**证据**：Free-code的70%自动压缩阈值可能过于保守——当70%的上下文窗口已被使用时，LLM可能已经在最近信息上失去连贯性了。

#### 过度依赖单一LLM提供商

提供商锁定产生脆弱性：
- **free-code**："工具与Anthropic API格式深度集成。支持其他LLM API需要可能丢失语义保真度的翻译层。"
- **OpenCode**：尽管支持15+个提供商，Vercel AI SDK引入了自己的依赖风险（beta版本）。
- **Pi**：通过清晰的抽象支持22+个提供商，但支持的质量因提供商而异。

教训：**从第一天起就设计为提供商独立**。LLM格局变化迅速，被锁定在一个提供商上是一种战略风险。

#### 忽视人机协作

在没有人类监督的情况下运行的Agent存在以下风险：
- 做出未被检测到的错误更改。
- 在错误的方法上浪费资源。
- 违反安全策略。

所有项目都包含某种形式的人机协作：
- Free-code的权限系统带有"ask"模式。
- OpenCode的Bus中介的权限请求。
- Pi的交互模式带有steering/follow-up队列。

陷阱不是缺乏人机协作，而是**粒度过粗**——如果每个Shell命令都需要批准，Agent变得无法使用。如果都不需要，它变得危险。找到正确的粒度依赖于上下文，需要仔细的设计。

### (o) 未来方向

分析揭示了Agent系统可能演化的几个方向：

#### 语义上下文管理（超越Token计数）

当前系统通过计算token来管理上下文。这是一个**粗糙的代理**，替代信息内容。一个100 token的关于关键架构决策的摘要比10,000 token的工具输出更有价值。未来的系统应该：
- 使用嵌入来测量语义相关性，而不仅仅是token计数。
- 优先保留与上下文中已有内容语义上距离较远的信息（最大化信息多样性）。
- 使用LLM自身的注意力模式来识别哪些上下文元素对决策最有影响。

#### 多模态工具接口

当前工具是文本输入、文本输出。未来的工具可以包括：
- 截图捕获和分析。
- 图表生成和解读。
- 用于语音命令的音频转录。
- 上下文中的结构化数据可视化。

OpenCode和Pi缺乏原生的视觉支持。Free-code有实验性的Web浏览器工具。这一差距将迅速缩小。

#### 跨会话的持久Agent记忆

当前的Agent在会话之间是失忆的——每个新会话都从头开始。未来的系统应该：
- 跨会话维护一个持久的知识图谱。
- 随时间学习项目特定的惯例和偏好。
- 建立一个成功模式库并重用它们。
- 记住哪些方法有效、哪些方法失败，跨会话和项目。

#### Agent到Agent的通信协议

子Agent当前通过父Agent的上下文窗口或通过文件系统产物进行通信。未来的系统应支持：
- Agent之间的直接消息传递。
- 具有事务语义的共享状态总线。
- 针对冲突变更的协商协议。
- 带有置信度评分的结果聚合。

OpenCode的Bus（PubSub）是这一模式的早期实现。

#### 自我修改的系统提示

系统提示目前每个会话是静态的。未来的系统可以：
- 基于任务上下文动态调整系统提示。
- 学习哪些提示组件对不同任务类型最有效。
- 从提示中裁剪未使用的技能和工具定义以节省token。
- 根据LLM展示的能力调整提示的语气和细节程度。

Pi的`transformContext`钩子暗示了这一方向，但尚未完全自动化。

## 第五部分：结论

### (p) 研究发现摘要

本研究通过一个核心假设的透镜分析了三个编程Agent框架——free-code（Claude Code）、OpenCode和Pi——即Agent框架从根本上用于将非马尔可夫的大规模项目分解为LLM可通过类bash接口消费的马尔可夫状态。

研究发现：

1. **假设大体正确但不完整。** 三个项目都投入了重大的架构精力来管理马尔可夫LLM（有限上下文窗口，调用之间无状态）与非马尔可夫项目（无限状态空间，深层历史依赖）之间的根本性阻抗失配。分解是真实的、有意的，并且在架构上被具体体现——不是偶然的属性。

2. **分解通过多个协调策略实现：** 上下文压缩（将对话历史转换为摘要）、工具媒介的状态访问（按需查询项目而非预加载它）、外部状态存储（会话持久化，文件系统作为基础事实）以及任务分解（子Agent、待办列表、会话分支）。没有单一策略是充分的；是它们的组合才奏效。

3. **类bash接口是LLM的符号推理与确定性外部世界之间的通用桥梁。** 它在每个项目中都存在，是继文件读/写之后最重要的工具。然而，它不是唯一的桥梁——专用工具（LSP、AST-grep、浏览器自动化）为特定领域提供了更高效、更可靠的接口。

4. **Agent框架并不消除非马尔可夫性；它管理它。** 完整的项目历史仍然存在（在文件系统、git历史、会话文件、数据库中）。LLM只是不一举持有全部。Agent框架提供了一个透镜，LLM通过它一次一个切片地观察项目并对之采取行动。

5. **上下文工程是核心学科。** 进入上下文窗口的内容决定了LLM能做什么。系统提示设计、工具定义策展、压缩质量和相关性过滤不是实现细节——它们是Agent系统的核心设计问题。

6. **三层架构（外部状态 → Agent运行时 → LLM上下文窗口）和通用Agent循环（输入 → 提示 → 上下文 → LLM → 解析 → 执行 → 集成 → 压缩 → 重复）是趋同演化。** 它们源于相同的基本约束，出现在每个项目中，无论其起源、团队规模或设计哲学如何。

### (q) 对假设的最终裁决

**原始假设：**

> Agent框架将非马尔可夫的大规模项目分解为纯马尔可夫状态，通过类bash接口提供，供GPT型语言模型操作。

**裁决：部分正确，修订如下：**

该假设正确地识别了核心设计问题（马尔可夫LLM与非马尔可夫项目之间的阻抗失配）和主要的解决模式（通过工具和压缩进行状态分解）。然而，它在三个方面过于简化了：

1. **"纯马尔可夫状态"是一种过度陈述。** LLM的上下文窗口仍然包含对话历史——一种违反严格马尔可夫性的情景记忆形式。Agent创建的是**近似马尔可夫的决策表面**，而非纯马尔可夫状态。

2. **"类bash接口"是必要的但非充分的。** Bash是通用后备，但特定领域的工具对于效率和可靠性至关重要。真正的创新是**结构化的动作空间**（具有确定性、幂等操作的工具），而非任何特定的接口。

3. **Agent所做的不仅仅是转换状态。** 它跨多个Agent分解任务、管理并行执行、强制权限、处理错误，并提供人类监督机制。状态转换是基础，而非整个建筑。

**修正后的论题：**

> Agent框架通过在非马尔可夫世界之上提供一个结构化的动作空间（确定性工具）、管理观察空间（上下文压缩和策展）以及外部化持久状态（文件、数据库、会话存储），来创建一个**马尔可夫决策过程（MDP）叠加层**。LLM仍然是一个马尔可夫状态机。Agent框架创造了使马尔可夫推理在非马尔可夫世界中足以实现有效行动的条件。

### (r) 宏观图景：Agent框架作为LLM的操作系统

本研究最深刻的洞见是：Agent框架正在演化为**LLM的操作系统**。类比是深层且实质性的，而非表面的：

| 操作系统概念 | Agent框架等价物 |
|-----------|---------------------------|
| **进程** | Agent会话 |
| **内存（RAM）** | LLM上下文窗口 |
| **磁盘 / 文件系统** | 会话存储、git仓库 |
| **I/O子系统** | 工具系统（bash、LSP、MCP） |
| **调度器** | Agent循环、子Agent创建器 |
| **虚拟内存** | 上下文压缩（盈余页面 → 磁盘） |
| **缺页中断** | 工具调用（从磁盘获取缺失状态） |
| **权限模型** | 工具权限评估器 |
| **设备驱动程序** | MCP服务器、专用工具 |
| **用户空间 / 内核空间** | LLM决策 / Agent运行时执行 |
| **进程分叉** | 带有上下文继承的子Agent创建 |
| **守护进程 / 服务** | 持久会话 |
| **文件描述符** | 上下文中的打开文件句柄 |

在三层架构框架下，马尔可夫分解论题就是Agent操作系统的**虚拟内存系统**。正如虚拟内存通过将页面换出到磁盘来创造进程拥有比物理RAM更多内存的幻觉，Agent上下文管理通过压缩旧状态并按需获取新状态，创造了LLM拥有比其token限制更多上下文的幻觉。

类bash接口是**系统调用接口**——一小部分稳定的、通用的操作，用户空间程序（LLM推理）可以使用它们与内核管理的资源（文件、进程、网络）进行交互。正如Unix因其系统调用接口小而可组合而成功一样，Agent框架之所以成功，是因为bash在LLM推理与系统操作之间提供了一个小而可组合的接口。

Agent框架的未来是操作系统的未来：更好的资源管理、更高效的上下文利用、更丰富的I/O抽象，以及推理层（LLM）与执行层（Agent运行时）之间更深度的集成。本文分析的三个项目——free-code、OpenCode和Pi——是一种架构模式的早期实现，这种模式将像操作系统对计算一样，成为AI辅助软件工程的基础。
